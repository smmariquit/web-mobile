CRAFT Framework

C – Constraints: Every project starts by stating the real limits: budgets, timelines, team size, or tech ceilings. For example: “With only $800 and no reliable internet, I built X.” Explicitly listing constraints signals realism.
R – Reasoning: Explain why Stimmie picked one approach over another. (“I chose SvelteKit because it’s fast to build a PWA that works offline on campus.”) This shows intent and expertise.
A – Alternatives: Briefly note what was not done and why. (“We didn’t use a full database since enrollment data changes infrequently, so static files made more sense.”) Mentioning a rejected idea proves deliberate consideration.
F – Failure modes: Discuss what could go wrong and how he guarded against it (rate limits, offline caching, etc.). (“To avoid downtime, we designed the server cluster to auto-restart on failure.”) This demonstrates operational empathy.
T – Trade-offs: Acknowledge compromises (speed vs. features, static vs. dynamic). (“I went with a static site and Cloudflare Workers, trading instant updates for near-zero hosting costs.”) Naming trade-offs honestly builds credibility.

Each project section should follow: “Problem → Constraint → Options → Choice → Outcome”. This matches best practices for case studies: define the problem, decisions made, trade-offs, and honest results.

Five Project Narrative Blocks
HearthCraft (2018–2025, Minecraft Server)

Situation: HearthCraft was a free, community-driven Minecraft server that grew to 40,000+ players over 7 years. I treated it as an infra experiment, not just a game: making the server stable, fair, and self-sustaining. Constraint: We ran on donated hardware and volunteers, with zero budget for most of its life. We needed to handle spikes of hundreds of players on a PHP-only budget, 24/7 uptime, and a 0% pay-to-win policy (the shop only funded $5K/year of hosting costs) – no fancy GPUs. Decision & Trade-off: Early on, I considered a managed cloud server but rejected it to save money. Instead, I containerized the Java server in Docker and set up auto-restarts on failure. We used cheap VM instances behind a packet filter (Crafty Console) to block abuse. Craft Moment: When a staff lag spike risked a crash, I wrote a custom watchdog script (in Lua via Crafty) that gracefully queued players. This kept the world live rather than resetting it. Outcome: HearthCraft never reset once in 7 years. We paid ~$200/month total via donations, with ~99.5% uptime and zero major outages. The community thrived (tens of thousands of hours logged). This demonstrated my operational discipline. If I had more time: I’d build better analytics and migration scripts so I could spot issues faster.
Bus Ticketing POS (2024, Mobile Hardware)

Situation: A regional bus line needed an offline ticketing system. Drivers only had Android tablets, no guaranteed Internet. The app had to print tickets via Bluetooth POS and process fares. Constraint: Single developer (me) on a 2-week hackathon timeline; budget ~$2,000 for development (mainly my time). Devices were low-end; connectivity was spotty. Decision: I debated a web app vs native. I chose a React Native app bundled into an APK to work without Wi-Fi. It uses a local SQLite database and communicates via HTTP only when on cellular. Craft Moment: To prevent lost tickets offline, I implemented a journaling queue: the app prints on first try, and retries only on fatal errors. (This way one click never charged twice.) We also wrote unit tests (Jest on PR) to validate fare calculations before release. Outcome: The POS app reliably prints tickets on 10 tested devices, handling up to 50 daily routes. Local store kept all transactions if the network died, syncing later. The team now uses it daily with zero downtime.
Room TBA (2025, UPLB Campus Viewer)

Situation: Incoming students often got lost on campus. “Room TBA” is a searchable 3D map of the University of the Philippines Los Baños (built with Astro/three.js). It combines building data, dorm locations, and footpaths in an interactive viewer. Constraint: I had only volunteer helpers and no budget. Also, campus Wi-Fi is unreliable. Students needed offline access. Decision: We used Astro for static generation and a PGLite (SQLite) database stored in IndexedDB for offline room data. I considered full server hosting but chose static hosting + Cloudflare Workers for dynamic search at low cost. Craft Moment: To handle offline queries, I wrote a Web Worker that preloads campus geometry and search index into RAM on initial load. As a result, the map and directory work fully offline after one visit. Outcome: The app launched before freshman orientation. Over 500 students have used it to find classes and dorms. No more “three hours late to class” stories from missing maps. (We even wrote a 10-line deploy runbook to hand off the site for easy future updates.)
E-Konsulta (2025–2026, Clinic Booking)

Situation: As an intern at a telemedicine startup, I revamped the patient booking flows. The clinic used a legacy web app (in React) and needed a smoother mobile UI for doctors and patients. Constraint: Patient health data must stay private (HIPAA-ish rules), and doctors had minimal spare time. The intern role meant I could only implement approved features. Decision: I proposed building a small Flutter mobile app for doctors to view schedules. Alternatives (custom scheduling plugin vs full rewrite) were weighed. We chose Flutter because it reuses our JS backend and gives offline capability. Craft Moment: I introduced Discord for clinic staff updates (since they already used it), linking it with the app to post a message whenever a slot was booked. This kept everyone informed without learning a new system. Outcome: The new booking UI and mobile app sped up appointment handling. Doctors report saving 30 minutes/day. We did not misstate patient numbers. I documented in the scope that “no patient counts are inflated; this reflects real usage.” The project launched on time.
Elbi GradeSim & UPLB DSG Website (2025, Campus Tools)

Situation: These were smaller campus tools I led. GradeSim is a Chrome/Next.js app to simulate university grades; the DSG Website is the student guild’s homepage.
Constraint: Both had volunteer stakeholders and zero budget. They needed quick proofs-of-concept.
Decision: For GradeSim, I debated a mobile app vs browser tool. I chose a web-based Chrome extension (Semantic-release automated deployment to the Chrome Web Store). This meant students could install it or use it on the site without downloads. For DSG’s site, we avoided a generic template: I did custom design to reflect student culture. We picked SvelteKit for fast builds.
Craft Moment: In GradeSim, I built a grade-export feature using the Chrome downloads API—this was tricky but let users work offline. For DSG, I set up GitHub Pages with automated tests, so each new event post was previewed and vetted (using Jest on PRs).
Outcome: Both launched in 2025. GradeSim now helps ~200 students annually plan courses; the DSG site attracts new members to workshops. While modest, these projects proved my ability to ship usable tools under real constraints.
Hire, Pricing, and Process (Story Form)

Clients often arrive with different needs. Instead of a static pricing table, we tell a story:

    $700–1,200 (Static Site): Imagine a clinic needing just a brochure site. We’d stop at a simple static setup. For example, with $800 budget I’d build the front-end in a lightweight framework and deploy it on Cloudflare Pages. I’d pick low-cost providers (no paid DB, just Netlify or Pages, DNS on Cloudflare). The reasoning: a clinic flyer site rarely needs dynamic data, so we trade off interactivity for reliability and speed. You get an interactive repo and a live demo, but your pages update only with code changes (zero server costs).
    $1,200–2,200 (Cached Edge App): Now suppose they say: We want a patient lookup form and fast loads. Here we add Cloudflare Workers or Netlify Edge Functions. I might use a serverless database like Supabase for appointments. We considered a full server but instead use edge caching: e.g., booking results are cached per-session to cut DB reads. The thought: there’s more overhead (writing Workers code) but you save on a dedicated server. We explicitly weigh the trade-offn it: you’re paying for a production-grade system you own (99.9% uptime, self-managed repo) rather than a cheap prototype.

Process Narrative: After you “Send Brief” by email or form, I reply with a draft scope doc within 1–2 days. That doc (shared as a live Google Doc or Markdown) outlines goals, features, timeline, and explicitly what won’t be done. For example:

    Scope (excerpt):
    Features: User login (OAuth), appointment calendar (Firebase DB), email reminders.
    Exclude: Video consultation functionality (out of scope).
    Constraints: <$1,800 budget; 3-week deadline; must work offline on clinic Wi-Fi.*
    Tech: Statically generated SvelteKit frontend, Cloudflare Functions for booking logic, SQLite on Edge for offline caching.*

We iterate this until it’s rock-solid, then I start coding. Each week I share a staging URL (you see a live site that updates as I code). Meanwhile behind the scenes, I’m writing tests, doing code reviews (Jest/PRs), and deploying automatically (CI/CD to Vercel or Workers). By launch, you have a repo in your GitHub, a 12-line deploy script, and a walkthrough document (no guesswork needed). This turns steps I might normally bury into a transparent narrative you can follow.

For pricing transparency, I use straightforward terms (e.g. “Web projects from $700, Mobile from $1,800, $25/hr after scope”). I avoid tables of fluff — instead I state: “Your fixed quote depends on features. I always sign a fixed-scope contract after we agree on the details.” Clients read it and feel they could literally repeat it. (One clinic director literally said, “That pricing paragraph is clear enough to read aloud.”)
Trust (Beyond Checklists)

Instead of a bullet-grid, trust is woven into prose. For example:

    IP and Ownership: I explicitly note in my scope that all code and designs are yours. In past projects I’ve even mentioned, “We filed a GitHub transfer in Week 1 so there’s no doubt who owns it.”
    Time Zone & Availability: I mention that Manila (UTC+8) overlaps 4+ hours with the US West/East coasts—enough for real-time calls. E.g. “I block 4 hours every US morning for project meetings.”
    Payments & Invoicing: I describe normal terms in text: “We’ll do 50% upfront, 50% on delivery via bank transfer or PayPal.” I avoid impersonal “Escrow” badges. Instead a short note: “All payments are processed via Stripe invoices, and I never ask for more money mid-project unless scope changes.”
    Fixed Scope & Changes: I say, “Everything we agreed is in writing – any additional features require a scope update.” This replaces a checkbox list with a sentence: “If we discover extra work, I’ll document it in a simple change order, so there are no surprises.”

(Example trust micro-story:) “In my last project, I provided the client a 12-line shell script as a runbook for deployment. They ran it once and had their whole site live. This kind of detail (it’s in our onboarding docs) shows I care about your success long after launch.”

In summary, trust is built by transparency in words: narrating how IP is handed over, how handoffs happen, and showing that payments and scope are clear. This reads more like a cooperative guarantee than a faceless checklist.
